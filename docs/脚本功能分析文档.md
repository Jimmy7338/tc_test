# TC 脚本功能分析文档

## 1. 脚本文件与设备分工

当前先按现场新结论处理：四个脚本里只有顶扫 `tc_in_0401.js` 作为有效业务脚本分析和维护。侧扫脚本暂时不作为当前业务链路的一部分，`tc_out_0401.js` 与顶扫逻辑同源时也不重复分析。

`tc` 文件夹中相关文件可以按当前维护优先级理解为：

- `tc_in_0401.js`：当前有效顶扫输出脚本，负责输入快照日志、命令处理、metadata、ROI 过滤、条码分类、历史去重、输出决策和漏触发补偿。
- `tc_out_0401.js`：顶扫同类脚本，如果现场仍保持与 `tc_in_0401.js` 一致，可作为同步对象，不作为本次主分析对象。
- `tc_in_front_0401.js`、`tc_out_front_0401.js`：侧扫 ROI 选择脚本，当前先忽略，不纳入本次重构后的文档主链路。

重构后的 `tc_in_0401.js` 已移除顶扫脚本内残留的 `SetRoiIndex/RegisterCallback` 侧扫回调职责，文件末尾只保留顶扫入口 `processCodes()`。

整体上可以理解为：

- 顶扫脚本负责最终输出格式化、条码分类、历史去重、漏触发补偿和元数据拼接。
- 顶扫脚本内部使用 Context 流水线组织单次任务数据，输出决策拆成多个高内聚 handler。
- 如果现场仍配置了侧扫 ROI 状态，顶扫脚本会读取 `VNLib.GetSlaveRoiIndex()` 做 ROI 过滤；如果侧扫链路不再使用或无返回，则 ROI 过滤会自动跳过。

## 2. 运行环境和脚本依赖

这些脚本运行在下位机 JS 脚本引擎中，依赖系统预注入的全局对象和变量。

### 2.1 依赖的全局对象

脚本大量使用 `VNLib` 和 `GlobalString`：

- `VNLib.Log(msg)`：输出脚本调试日志。
- `VNLib.SendOutput(text)`：向外部输出格式化后的扫码结果。
- `VNLib.getSeparator()`：获取多码结果之间的分隔符。
- `VNLib.GetMissedTriggerCountDuringTask()`：获取当前任务期间漏触发数量，用于补偿输出。
- `VNLib.GetBoxCoordinates()`：获取包裹检测框坐标。
- `VNLib.GetBoxLineCoordinates()`：获取虚拟检测线坐标。
- `VNLib.GetBoxDirection()`：获取输送方向，0 左到右，1 右到左，2 上到下，3 下到上。
- `VNLib.GetOnlineSlaveCount()`：获取在线从机数量。
- `VNLib.GetSlaveRoiIndex()`：获取侧扫 ROI 模式信息。
- `VNLib.GetRoiPoints()`：当前顶扫主流程不直接使用，侧扫脚本中才会用于 ROI 框读取。
- `VNLib.RegisterCallback(name, func)`：当前顶扫脚本不再使用；侧扫脚本如果重新启用时才需要关注。

`GlobalString` 用于跨次任务保存状态：

- `GlobalString.load(key)`：读取持久字符串变量。
- `GlobalString.store(key, value)`：写入持久字符串变量。

### 2.2 依赖的脚本变量

顶扫脚本要求格式配置中注入以下变量：

- `code`：当前任务识别到的所有码内容，多个码之间用 `VNLib.getSeparator()` 分隔。
- `center`：每个码的中心点坐标，顺序与 `code` 对齐。
- `ROI_number`：每个码对应的 ROI 编号，顺序与 `code` 对齐。
- `time`：任务时间，用于输出 `Metadata.Time`。
- `device_number`：设备编号，当前脚本默认不输出到 metadata，但会记录日志。
- `strStored`：串口收到的外部命令。
- `strTcpStored`：TCP 收到的外部命令。
- `box_coordinate`、`box_angle`、`is_box_pass_line`：包裹相关信息，脚本中会记录日志或重新计算。

注意：当前顶扫脚本不再注册侧扫回调。部署时重点确认目标设备版本支持顶扫所需的 `VNLib.SendOutput()`、`VNLib.GetMissedTriggerCountDuringTask()`、包裹坐标/虚拟线接口、在线设备数量接口和 `VNLib.GetSlaveRoiIndex()`。

## 3. 顶扫脚本总体业务目标

`tc_in_0401.js` 和 `tc_out_0401.js` 的核心目标是把设备识别结果转换成客户需要的 `Camera.PostScan` 输出格式。

输出字段结构为：

```text
Camera.PostScan    JobId    1Z/特殊一维码    Maxicode    PostalCode    DisposalMark    0    Metadata...
```

字段之间使用 `\t` 制表符分隔，行尾使用 `\r\n`。

脚本主要完成以下事情：

1. 首行输出 `InputSnapshot`，聚合记录本次任务的输入变量和设备状态，便于现场排查。
2. 接收并处理外部命令，例如 `bypass on`、`bypass off`、`resetnum`。
3. 生成任务流水号 `JobId`。
4. 从码内容中识别不同码制类型：`1Z`、Maxicode、PostalCode、特殊一维码。
5. 根据侧扫 ROI 模式过滤条码；如果 ROI 状态不可用，则跳过过滤。
6. 根据业务规则生成 `DisposalMark`。
7. 处理历史去重和超时清理。
8. 处理 Maxicode 中嵌入的特殊一维码。
9. 生成包裹角度、时间、过线状态、相机在线状态、箱型、边长等元数据。
10. 根据漏触发数量补偿输出额外 `PostScan` 行。
11. 通过拆分后的输出决策 handler 按优先级选择最终输出分支。

## 4. 顶扫脚本模块说明

### 4.1 配置开关模块

脚本开头定义了若干开关：

- `enableSerialResponse`：是否对外部命令返回应答。
- `debugmode`：调试输出方式，0 表示只写日志，1 表示输出到外部，2 表示日志和外部输出都做。
- `enableTimeoutClear`：是否启用历史条码超时清理。
- `timeoutSeconds`：历史条码超时时间，当前为 10 秒。
- `enableNoBarcodeOutput`：无条码时是否不输出。当前为 `false`，即无条码也输出 `????`。
- `useTimeInMetadata/useAngleInMetadata/...`：控制 metadata 中包含哪些字段。
- `CODE_PLACEHOLDER`：普通缺失字段占位符，当前为 `????`。
- `CONFLICT_PLACEHOLDER`：同类多码或多包裹冲突占位符，当前为 `!!!!`。
- `DISPOSAL_NORMAL/DISPOSAL_INCOMPLETE/DISPOSAL_MULTI_PACKAGE/DISPOSAL_SPECIAL_1D`：集中定义 `DisposalMark` 含义，避免各分支硬编码难维护。

这些开关决定脚本运行时的输出行为和日志行为。

### 4.1.1 Context 与首行输入日志

重构后脚本使用 `createRuntimeContext()` 作为单次任务的数据载体，集中保存：

- `input`：脚本注入变量，例如 `code`、`center`、`ROI_number`、`time`、串口/TCP 命令、包裹信息。
- `deviceNumbers`、`boxAngle`：设备号和包裹角度。
- `metadata`：最终输出 metadata 字符串。
- `filteredCodeStr/filteredCenterStr`：ROI 过滤后的原始字符串。
- `codes/centerArr`：拆分后的条码和中心点数组。
- `previousOutput`：历史输出主码。
- `classified`：四类条码数组。
- `codePositions`：条码到中心点坐标的映射。
- `disposalMark`、`historyCleared`：输出决策状态。

脚本进入 `processCodes()` 后第一条日志是 `InputSnapshot`，会聚合输出：

```text
InputSnapshot | version=... | code=... | center=... | ROI_number=... | time=... | strStored=... | strTcpStored=... | box_coordinate=... | device_number=... | box_angle=... | is_box_pass_line=...
```

排查现场问题时，建议优先看这条日志，确认上位机变量映射和设备注入值是否符合预期。

### 4.2 外部命令处理模块

`handleSerialMessage(message)` 处理 `strStored` 或 `strTcpStored` 中的命令。

支持命令：

- `bypass on`：开启 bypass 模式，并保存 `GlobalString["bypassMode"]="true"`。
- `bypass off`：关闭 bypass 模式，并保存 `GlobalString["bypassMode"]="false"`。
- `resetnum`：把 `jobIdCounter` 重置为 0。

命令来源优先级：

1. 如果 `strStored` 有值，优先处理 `strStored`。
2. 否则处理 `strTcpStored`。

业务意义：

- 现场可通过串口或 TCP 控制脚本状态。
- `resetnum` 用于重置输出流水号。
- bypass 模式下输出任务号固定为 `00-1`。

注意：旧版本代码中曾出现过初始化 bypass 反向的问题：

```js
let bypassMode = GlobalString.load("bypassMode") === "true";
```

重构后应保持 `GlobalString["bypassMode"]="true"` 对应 bypass 开启，`"false"` 对应普通模式。现场如果发现 bypass 状态和命令语义相反，需要优先检查该初始化逻辑是否被改回旧写法。

### 4.3 Metadata 生成模块

顶扫脚本会组合输出 metadata，当前包含：

- `Metadata.Skewness`：包裹偏斜角，由 `getParcelAngle()` 计算。
- `Metadata.Time`：时间，使用 `convertToISO8601Zulu(time)` 转成 Zulu 格式。
- `Metadata.Distance`：包裹是否压到/触碰检测线，由 `checkBoxAndLineIntersection()` 计算。
- `Metadata.Camera.Status`：相机在线数量，由 `GetDeviceStatus()` 计算。
- `Metadata.Box`：侧扫判断出的箱型模式，`Tall` 或 `Short`。
- `Metadata.boxlength`：包裹边长，由 `getParcelEdgeLength()` 计算。

metadata 字段之间用两个空格分隔。

### 4.4 JobId 生成模块

`getNextJobId()` 负责生成任务号：

- bypass 模式下固定返回 `00-1`。
- 普通模式下从 `GlobalString["jobIdCounter"]` 读取上次值，加 1 后保存。
- 计数对 9996 取模，0 会被改成 1。
- 正常输出为 4 位字符串，例如 `0001`、`0002`。

这个模块保证不同任务输出有连续编号，同时允许通过 `resetnum` 命令重置。

### 4.5 输出封装模块

`formatOutput(...)` 负责组装最终输出行：

```text
Camera.<messageType>    jobId    oneZCode    maxicodeData    postalCode    disposalMark    0    metadata
```

`sendOutput(str)` 调用 `VNLib.SendOutput(str + endStr)` 输出结果。

如果输出内容以 `Camera.PostScan\t` 开头，`sendOutput` 还会调用 `buildMissedTriggerCompensationOutput`，根据漏触发数量追加补偿行。

### 4.6 漏触发补偿模块

`buildMissedTriggerCompensationOutput(lastPostScanLine)` 使用 `VNLib.GetMissedTriggerCountDuringTask()` 获取任务期间漏触发数量。

如果漏触发数量大于 0，会额外输出多行：

```text
Camera.PostScan    新JobId    ????    ????    ????    0    0    Metadata.Skewness=???? ...
```

业务意义：

- 当硬件 IO 有触发但没有形成完整识别结果时，用补偿行占位。
- 补偿行同样消耗 JobId，保证包裹数量和输出行数对齐。

### 4.7 ROI 过滤模块

`filterCodesByRoi(codeStr, centerStr, roiNumberStr)` 会根据侧扫 ROI 模式过滤顶扫识别到的码。

流程：

1. 调用 `judgeRoiMode()`。
2. `judgeRoiMode()` 通过 `VNLib.GetSlaveRoiIndex()` 获取侧扫 ROI 状态。
3. 如果任一侧扫返回值为 1，则认为是 `Tall`。
4. 否则认为是 `Short`。
5. `Tall` 对应目标 ROI 索引 2。
6. `Short` 对应目标 ROI 索引 1。
7. 遍历 `code/center/ROI_number`，只保留 ROI 编号等于目标索引的码。

如果 ROI 模式有效且原始条码存在，但没有任何码匹配目标 ROI，当前脚本会返回空 `code/center`，让后续流程进入无码输出分支，最终按 `????` 输出，而不是回退输出原始条码。

业务意义：

- 如果系统仍提供 ROI 模式，顶扫根据该模式只输出对应 ROI 区域里的码。
- 如果 ROI map 中任一值为 `1`，顶扫按 `Tall` 处理；否则按 `Short` 处理。
- 如果当前现场不再使用侧扫链路，`VNLib.GetSlaveRoiIndex()` 返回为空或无法解析时，顶扫脚本会跳过 ROI 过滤。

### 4.8 历史去重和超时清理模块

脚本使用两个全局变量保存历史：

- `lastTaskCodes`：最近输出过的主码历史。
- `codeTimes`：每个码第一次记录的时间。

`checkAndClearTimeoutHistory()` 会在每次处理前检查历史记录，超过 `timeoutSeconds` 的码会被清理。

`storeHistory(codes)` 会把当前输出的主码写入历史，最多保存 4 条记录。

`isDuplicate(code)` 判断某个码是否已经存在历史中。

业务意义：

- 避免同一个包裹在多次触发/多帧识别中重复输出。
- 10 秒后自动释放历史，避免长期误判为重复。

### 4.9 条码分类模块

`processCodes()` 会把当前码分成几类：

- `qrCodes`：以 `[)` 开头，认为是 Maxicode。
- `oneZCodes`：以 `1Z` 开头。
- `postalCodes`：长度等于 8，认为是邮编码。
- `specialOneDCodes`：以 `B` 或 `1B` 开头的特殊一维码。

新版本关键点：

- 特殊一维码和 1Z 被合并视为 “1D-like” 类别。
- 多包裹判断前会先用历史记录过滤掉已经输出过的码。
- 如果历史过滤后的 `1Z + 特殊一维码` 数量大于 1，认为是多包裹。
- 如果 Maxicode 数量大于 1，也认为是多包裹。

### 4.10 DisposalMark 判定模块

`getDisposalMark(oneZCodes, specialOneDCodes, maxiCodes, postalCodes)` 返回处置标记：

- `0`：正常单包裹，通常是 1 个 1Z/特殊一维码 + 1 个 Maxicode。
- `1`：不完整或无有效主码，例如只有 1Z、只有 Maxicode、只有邮编、没有主码。
- `2`：多包裹或同类多码冲突，例如多个 Maxicode，或 `1Z + 特殊一维码` 总数大于 1。
- `3`：特殊一维码相关输出。脚本中当发现特殊一维码或从 Maxicode 中解析出特殊一维码时，会使用 3。

### 4.11 Maxicode 特殊一维码解析模块

`extractSpecialOneDFromMaxicode(maxicode)` 会解析 Maxicode 字段：

1. 去掉开头 `[)>`。
2. 将可视化控制字符 `␝/␞` 替换为真实 ASCII 控制字符 GS/RS。
3. 按 GS/RS 拆分字段。
4. 读取第 5 个非空字段。
5. 如果字段以 `B` 或 `1B` 开头，则认为其中包含特殊一维码。

业务意义：

- 某些场景下只有 Maxicode 被读到，但 Maxicode 内部实际携带一个特殊一维码。
- 脚本会把这个特殊一维码识别出来，并把 `DisposalMark` 改成 3。

### 4.12 主业务流程 `processCodes()`

重构后主流程采用 Context 流水线结构：先把本次任务的注入变量、设备状态、metadata、过滤结果、分类结果、历史状态和最终输出决策集中在一个上下文对象里，再按阶段逐步处理。

主流程如下：

1. `createRuntimeContext()`：读取注入变量和设备状态，生成本次任务上下文。
2. `logInitialInputSnapshot()`：第一行输出所有输入信息。
3. `handleConfiguredCommand()`：按 `strStored > strTcpStored` 优先级处理外部命令。
4. `buildMetadata()`：独立生成 metadata。
5. `validateRuntimeInput()`：校验 `code/center`，缺失时按无码输出兜底。
6. `checkAndClearTimeoutHistory()`：清理超时历史。
7. `applyRoiFiltering()`：根据 `judgeRoiMode()` 和 `ROI_number` 过滤顶扫条码。
8. `splitFilteredCodes()`：拆分条码和中心点。
9. `loadPreviousOutput()`：读取历史主码。
10. `classifyCodes()`：分类为 Maxicode、1Z、PostalCode、特殊一维码。
11. `calculateDisposalMarkForContext()`：历史过滤后计算 `DisposalMark`。
12. `buildCodePositions()`：建立条码到中心点坐标的映射。
13. `decideAndEmitOutput()`：按业务优先级选择输出分支。

### 4.13 输出决策 handler

`decideAndEmitOutput()` 已重构为规则调度入口，顺序如下：

1. `handleSpecialOneDOutput(data)`：特殊一维码最高优先级；选择最近未输出特殊一维码，并尽量匹配最近 PostalCode 和 Maxicode，输出 `DisposalMark=3`。
2. `handleIncompleteOutput(data)`：处理 `DisposalMark=1` 的不完整结果，包括单特殊一维码、单 1Z、单 Maxicode、只有 PostalCode、完全无码。
3. `DISPOSAL_MULTI_PACKAGE` 分支：多包裹或同类多码冲突，输出 `!!!!/!!!!/!!!!` 和 `DisposalMark=2`。
4. `handleContainedPairOutput(data)`：检查当前任务内 1Z 与 Maxicode 的包含关系，匹配成功时组合输出。
5. `handleClosestMainCodeOutput(data)`：没有完整组合时，选择距离坐标原点最近的未输出 1Z 或 Maxicode 单独输出。
6. `handlePostalFallbackOutput(data)`：没有新主码但存在 PostalCode 时，根据主码历史状态决定输出 PostalCode 或 `????`。
7. 最终兜底：输出 `????/????/????`。

输出相关的公共 helper：

- `emitPostScan()`：统一正式 `Camera.PostScan` 输出入口。
- `emitPlaceholderPostScan()`：统一无码/占位输出入口。
- `findClosestCode()`、`findClosestPostal()`、`findClosestMaxicode()`：统一最近距离选择逻辑。
- `storeHistoryAndLog()`：统一历史写入和日志。
- `oneZBlockedByHistory()`、`maxicodeBlockedByHistoryOneZ()`、`hasMaxicodeHistoryContainment()` 等：集中处理历史包含关系判断。

## 5. 当前暂不分析的侧扫脚本

本次文档按用户要求先忽略侧扫代码。`tc_in_front_0401.js` 和 `tc_out_front_0401.js` 暂不作为当前有效业务脚本展开说明。

对当前顶扫脚本来说，需要知道的侧扫相关点只有一个：

- 如果系统仍向顶扫暴露 `VNLib.GetSlaveRoiIndex()`，顶扫会据此判断 `Tall/Short` 并过滤 ROI。
- 如果当前现场侧扫链路已经不用，`VNLib.GetSlaveRoiIndex()` 返回为空、不可解析或没有 ROI map 时，顶扫会跳过 ROI 过滤，继续处理原始顶扫条码。

重构后的 `tc_in_0401.js` 内部不再定义或调用 `SetRoiIndex/RegisterCallback`。

## 6. 当前顶扫数据链路

当前有效链路可以按以下方式理解：

1. 设备完成顶扫识别并向脚本注入 `code/center/ROI_number/time` 等变量。
2. 顶扫脚本创建 RuntimeContext，并首行输出 `InputSnapshot`。
3. 脚本处理串口/TCP 命令，例如 bypass 和 resetnum。
4. 脚本生成 metadata。
5. 如果有有效侧扫 ROI 状态，则按 `Tall/Short` 过滤顶扫条码；否则跳过 ROI 过滤。
6. 脚本拆分条码、读取历史、分类、计算 `DisposalMark`。
7. 脚本按 handler 顺序选择输出分支。
8. 脚本通过 `VNLib.SendOutput()` 输出 `Camera.PostScan`。
9. 如果当前任务有漏触发计数，脚本追加补偿 `PostScan` 行。

## 7. 业务场景举例

### 7.1 正常单包裹

识别到：

- 一个 1Z 条码。
- 一个 Maxicode。
- 可选 PostalCode。

脚本输出：

```text
Camera.PostScan    0001    1Z...    [)>...    postal    0    0    Metadata...
```

### 7.2 只读到 1Z

脚本认为结果不完整，输出：

```text
Camera.PostScan    0001    1Z...    ????    postal或????    1    0    Metadata...
```

### 7.3 只读到 Maxicode

脚本会先检查 Maxicode 内是否有特殊一维码：

- 如果没有特殊一维码，输出 Maxicode，1Z 字段为 `????`，`DisposalMark=1`。
- 如果解析到特殊一维码，`DisposalMark` 改为 3。

### 7.4 同类多码或多包裹

如果历史去重后：

- `1Z + 特殊一维码` 数量大于 1，或
- Maxicode 数量大于 1，

脚本输出：

```text
Camera.PostScan    0001    !!!!    !!!!    !!!!    2    0    Metadata...
```

### 7.5 漏触发补偿

如果一次正常输出后，`VNLib.GetMissedTriggerCountDuringTask()` 返回 2，则脚本会额外输出 2 行补偿：

```text
Camera.PostScan    0002    ????    ????    ????    0    0    Metadata.Skewness=???? ...
Camera.PostScan    0003    ????    ????    ????    0    0    Metadata.Skewness=???? ...
```

## 8. 当前脚本中的注意点

### 8.1 首行 `InputSnapshot`

当前脚本第一行业务日志应该是 `InputSnapshot`。如果现场日志第一行不是该内容，需要确认是否粘贴了最新脚本，或脚本执行前是否已有其他外部日志。

### 8.2 顶扫脚本不再包含侧扫回调

`tc_in_0401.js` 作为顶扫输出脚本，不应再包含：

```js
function SetRoiIndex(...) { ... }
function RegisterCallback() { ... }
RegisterCallback();
```

如果在顶扫脚本中看到这些函数，说明使用的不是当前重构后的版本。

### 8.3 bypass 初始化逻辑

顶扫脚本应使用：

```js
let bypassMode = GlobalString.load("bypassMode") === "true";
```

这样 `bypass on` 写入 `"true"` 后会进入 bypass，`bypass off` 写入 `"false"` 后会恢复普通模式。

### 8.4 ROI 编号假设需要和下位机一致

顶扫脚本中写死：

- `Tall` 对应 ROI index 2。
- `Short` 对应 ROI index 1。

如果系统提供的 ROI map 与顶扫 `ROI_number` 的一基编号不一致，或者现场 ROI 编号不是 1/2，就可能导致顶扫过滤不符合预期。

### 8.5 顶扫脚本的“距离原点最近”不是“距离画面中心最近”

脚本中的 `calculateDistance(x, y)` 是：

```js
Math.sqrt(x * x + y * y)
```

也就是距离坐标原点 `(0,0)` 最近，而不是距离图像中心最近。  
如果业务要求是“距离画面中心最近”，需要改为相对图像中心计算。

### 8.6 输出决策顺序不能随意调整

`decideAndEmitOutput()` 的 handler 顺序是业务优先级：

1. 特殊一维码。
2. 不完整结果。
3. 多包裹/同类多码。
4. 1Z 与 Maxicode 包含关系。
5. 最近未输出主码。
6. PostalCode 兜底。
7. 无码兜底。

如果新增规则，需要明确插入位置，否则可能改变既有输出行为。

## 9. 建议验证项

建议现场按以下顺序验证：

1. 验证第一条业务日志是否为 `InputSnapshot`，并确认 `code/center/ROI_number/time` 等输入值正确。
2. 验证 `bypass on/off` 和 `resetnum` 命令是否按 `strStored > strTcpStored` 优先级生效。
3. 验证 `Camera.PostScan` 输出字段顺序是否符合客户协议。
4. 验证特殊一维码是否优先输出且 `DisposalMark=3`。
5. 验证单 1Z、单 Maxicode、只有 PostalCode、无码等不完整场景输出。
6. 验证多包裹场景是否输出 `!!!!` 且 `DisposalMark=2`。
7. 验证 1Z 与 Maxicode 包含关系能组合输出。
8. 验证没有完整组合时是否选择距离坐标原点最近的未输出主码。
9. 验证 10 秒后历史条码是否能重新输出。
10. 验证漏触发场景下是否追加补偿行。
11. 如果现场仍使用 ROI，验证 `ROI filter - roiMode` 和过滤后的条码数量；如果不使用 ROI，确认 ROI 不可用时不会阻塞输出。

## 10. 总结

这套脚本不是简单的格式化输出脚本，而是承担了较完整的现场业务决策：

- 当前有效脚本是顶扫 `tc_in_0401.js`。
- 顶扫脚本使用 Context 流水线整理单次任务数据。
- 顶扫脚本通过多个输出决策 handler 完成条码分类、历史去重、Maxicode 解析、多包裹判断、漏触发补偿和最终输出。
- 侧扫代码当前先忽略；顶扫仅在 `VNLib.GetSlaveRoiIndex()` 有有效返回时使用 ROI 过滤。

最终目标是让多相机组网系统在复杂包裹场景下，只输出符合当前包裹形态和客户协议的一条或多条 `Camera.PostScan` 结果。
